# Правильная доменная модель — Запись к специалисту

## Контекст задачи

Система позволяет пациентам записываться к специалистам клиники. Специалисты и администраторы могут управлять расписанием и отменять записи.

---

## Entities (Сущности)

Сущность — объект с уникальной идентичностью, который живёт во времени и меняет состояние.

| Сущность | Обоснование |
|----------|-------------|
| **Patient (Пациент)** | Имеет уникальный ID, живёт независимо от записей. Один и тот же пациент делает множество записей за всё время. |
| **Specialist (Специалист)** | Имеет уникальный ID, независимый жизненный цикл. Специалист существует в системе даже если у него нет записей. |
| **Appointment (Запись на приём)** | Имеет уникальный ID, меняет статус (создана → подтверждена → отменена → завершена). Это отдельный объект с историей. |
| **Schedule (Расписание)** | Содержит слоты специалиста на определённый период. Изменяется независимо (добавляются/блокируются слоты). Является **корнем агрегата**. |
| **Administrator (Администратор)** | Участник системы с особыми правами (может отменять записи без ограничений по времени). |

---

## Value Objects

Value object — объект без собственной идентичности. Два VO с одинаковыми полями — это одно и то же. Неизменяемый.

| Value Object | Обоснование |
|--------------|-------------|
| **PatientId** | Типизированный идентификатор. Используется как ссылка из других агрегатов вместо полного объекта Patient. |
| **SpecialistId** | То же самое для специалиста. Агрегаты ссылаются друг на друга только через ID, не через объекты. |
| **AppointmentId** | Типизированный идентификатор записи. |
| **ContactInfo** | Телефон или email пациента. Нет идентичности — два пациента с одним телефоном имеют одинаковый ContactInfo, но это разные пациенты. |
| **TimeSlot** | Пара (начало, конец) — конкретный временной интервал. Не имеет идентичности, определяется значением. Используется и в Schedule, и в Appointment. |
| **WorkingHours** | Начало и конец рабочего дня специалиста (например, 09:00–18:00). Описание правила, не факт. |
| **AppointmentStatus** | Перечисление: Created / Confirmed / CancelledByPatient / CancelledBySpecialist / Completed. Является частью Appointment. |

**Что НЕ является value object:**
- "Свободные слоты" — это вычисляемое состояние (результат метода Schedule), не хранимый объект
- "Записи пациента" — это коллекция сущностей, не VO

---

## Агрегаты и их границы

Агрегат — кластер объектов (сущностей и VO), у которых есть один корень (aggregate root). Всё взаимодействие с агрегатом идёт через корень. Один агрегат = одна транзакция.

---

### Агрегат 1: Appointment (Запись на приём)

**Корень:** Appointment

**Состав агрегата:**
```
Appointment (root)
  ├── AppointmentId          (VO — идентификатор)
  ├── PatientId              (VO — ссылка на пациента, не сам объект)
  ├── SpecialistId           (VO — ссылка на специалиста, не сам объект)
  ├── TimeSlot               (VO — забронированное время)
  └── AppointmentStatus      (VO — текущий статус)
```

**Что снаружи (не входит в агрегат):**
- Полный объект Patient — только ссылка PatientId
- Полный объект Specialist — только ссылка SpecialistId
- Расписание специалиста — отдельный агрегат
- Другие записи пациента — отдельные агрегаты

**Инварианты агрегата:**
- Отменённую запись нельзя подтвердить (только создать новую)
- Статусный переход: нельзя перейти из Completed назад в Confirmed
- Запись всегда создаётся в статусе Created

**Операции:**
```
Cancel(cancelledBy: PatientId, at: DateTime)
  → если cancelledBy == this.PatientId
     → проверить: now < slot.Start - 2 hours (иначе — отказ)
  → статус → CancelledByPatient

Cancel(cancelledBy: SpecialistId или AdministratorId)
  → без ограничения по времени
  → статус → CancelledBySpecialist

Confirm()
  → проверить: статус == Created
  → статус → Confirmed

Complete()
  → проверить: статус == Confirmed
  → статус → Completed
```

**Domain Events, которые публикует агрегат:**
- `AppointmentCancelled` — при любой отмене
- `AppointmentCreated` — при создании (используется для проверки конфликтов у пациента)

---

### Агрегат 2: Schedule (Расписание)

**Корень:** Schedule

**Состав агрегата:**
```
Schedule (root)
  ├── ScheduleId             (VO)
  ├── SpecialistId           (VO — чьё расписание)
  ├── WorkingHours           (VO — рабочее время, например 09:00–18:00)
  ├── WorkingDays            (VO — рабочие дни, например Пн-Пт)
  └── List<Slot>             (дочерние сущности — конкретные слоты)
        Slot
          ├── SlotId
          ├── TimeSlot       (VO)
          └── SlotStatus     (Free / Booked / Blocked)
```

**Что снаружи:**
- Полный объект Specialist — только ссылка SpecialistId
- Appointment — отдельный агрегат

**Инварианты агрегата:**
- Два слота не могут пересекаться по времени
- Слот не может выходить за рамки WorkingHours
- Заблокированный слот нельзя забронировать

**Операции:**
```
AddSlot(time: TimeSlot)
  → проверить: нет пересечений с существующими слотами
  → проверить: time внутри WorkingHours

BookSlot(slotId: SlotId, appointmentId: AppointmentId)
  → проверить: слот существует и статус == Free
  → статус слота → Booked

BlockSlot(slotId: SlotId)
  → проверить: слот существует
  → статус слота → Blocked
  → публикует: SlotBlocked(slotId, specialistId, timeSlot)

ReleaseSlot(slotId: SlotId)
  → статус слота → Free
```

**Domain Events:**
- `SlotBlocked` — при блокировке. Обработчик найдёт все Appointment на этот слот и отменит их через команду Cancel.

---

### Агрегат 3: Patient (Пациент)

**Корень:** Patient

**Состав агрегата:**
```
Patient (root)
  ├── PatientId              (VO)
  ├── Name                   (VO)
  └── ContactInfo            (VO — телефон или email)
```

**Что снаружи:**
- Записи пациента (Appointment) — отдельные агрегаты. Patient не хранит список своих записей внутри себя. Их запрашивают через AppointmentRepository по PatientId.

**Почему список записей не внутри Patient:**
Если пациент записан 100 раз — загружать всего пациента = загружать 100 записей. Это нарушает принцип маленьких агрегатов. История записей — отдельная проекция.

**Инварианты:**
- ContactInfo обязателен (нельзя создать пациента без контакта)

---

### Агрегат 4: Specialist (Специалист)

**Корень:** Specialist

**Состав агрегата:**
```
Specialist (root)
  ├── SpecialistId           (VO)
  ├── Name                   (VO)
  └── Specialty              (VO — специализация, например "Кардиолог")
```

**Что снаружи:**
- Расписание специалиста — отдельный агрегат Schedule
- Записи к специалисту — через AppointmentRepository

---

## Кросс-агрегатный инвариант: пациент не может записаться дважды в одно время

Этот инвариант нельзя проверить внутри одного агрегата, потому что он затрагивает несколько Appointment разных специалистов.

**Решение через Eventual Consistency:**

```
1. Appointment создаётся (нет проверки конфликта внутри агрегата)
2. Агрегат публикует: AppointmentCreated(patientId, timeSlot)
3. Domain Event Handler:
   a. Запрашивает все записи пациента за это время
   b. Если найден конфликт — отменяет новую запись с причиной "ConflictWithExistingAppointment"
   c. Уведомляет пациента
```

**Альтернатива — проверка в Application Service (более простая):**

```
CreateAppointmentHandler:
  1. Загрузить Schedule, проверить что слот свободен
  2. Запросить через AppointmentRepository: есть ли у пациента запись в это время?
  3. Если есть — выбросить исключение (не создавать)
  4. Создать Appointment
  5. Вызвать Schedule.BookSlot()
  6. Сохранить оба агрегата
```

Второй вариант проще, но требует явного написания в Application Layer, а не в домене.

## Диаграмма границ агрегатов

```
┌─────────────────────────────┐    ┌──────────────────────────────────┐
│  Aggregate: Schedule        │    │  Aggregate: Appointment          │
│  Root: Schedule             │    │  Root: Appointment               │
│  ─────────────────────────  │    │  ────────────────────────────    │
│  SpecialistId (ref)         │    │  AppointmentId                   │
│  WorkingHours               │    │  PatientId (ref)                 │
│  WorkingDays                │    │  SpecialistId (ref)              │
│  List<Slot>                 │    │  TimeSlot                        │
│    Slot                     │    │  AppointmentStatus               │
│      SlotId                 │    │                                  │
│      TimeSlot               │    │  Operations:                     │
│      SlotStatus             │    │    Cancel(by)                    │
│                             │    │    Confirm()                     │
│  Operations:                │    │    Complete()                    │
│    AddSlot()                │    └──────────────────────────────────┘
│    BookSlot()               │
│    BlockSlot() → [event]    │    ┌──────────────────────────────────┐
│    ReleaseSlot()            │    │  Aggregate: Patient              │
└─────────────────────────────┘    │  Root: Patient                   │
                                   │  ────────────────────────────    │
┌─────────────────────────────┐    │  PatientId                       │
│  Aggregate: Specialist      │    │  Name                            │
│  Root: Specialist           │    │  ContactInfo                     │
│  ─────────────────────────  │    └──────────────────────────────────┘
│  SpecialistId               │
│  Name                       │
│  Specialty                  │
└─────────────────────────────┘
```

**Стрелки зависимостей идут только через ID, никогда через объекты:**
- Appointment → Patient (через PatientId)
- Appointment → Specialist (через SpecialistId)
- Schedule → Specialist (через SpecialistId)

## Сводная таблица решений

| Концепция | Объект | Тип | Обоснование |
|-----------|--------|-----|-------------|
| Запись на приём | Appointment | Entity + Aggregate Root | Имеет ID, жизненный цикл (статусы), независимо изменяется |
| Временной слот | TimeSlot | Value Object | Определяется только значениями начало/конец, нет идентичности |
| Статус записи | AppointmentStatus | Value Object | Перечисление, нет идентичности |
| Слот расписания | Slot | Entity (внутри Schedule) | Имеет ID, меняет статус (Free/Booked/Blocked), но живёт только внутри Schedule |
| Рабочие часы | WorkingHours | Value Object | Описание правила, не факт; определяется значениями |
| Пациент | Patient | Entity + Aggregate Root | Уникальная идентичность, независимый жизненный цикл |
| Контакт пациента | ContactInfo | Value Object | Определяется значением телефона/email, нет идентичности |
| Специалист | Specialist | Entity + Aggregate Root | Уникальная идентичность, независимый жизненный цикл |
| Расписание | Schedule | Entity + Aggregate Root | Содержит инварианты (нет пересечений), владеет слотами |
| ID любого объекта | PatientId, SpecialistId и т.д. | Value Object | Типизированные VO для type safety |
