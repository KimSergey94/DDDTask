# Руководство по DDD: Агрегаты, корни и границы

## Для кого этот документ

Для тех, кто только начинает изучать DDD и путается в базовых понятиях: что такое агрегат, как найти корень, почему это не то же самое, что use case. Всё объясняется на примере системы записи к специалисту.

---

## Часть 1. Откуда берётся путаница

Самая частая ошибка новичков в DDD — проектировать агрегаты **от действий**:

> "Пациент записывается к врачу → значит, вот все данные, которые мне нужны для этого действия → это и есть агрегат"

Это неправильный ход мысли. Давайте разберём почему.

### Действие vs Объект

Представьте реальный мир. Есть карточка пациента. В ней: имя, дата рождения, контакт. Карточка существует сама по себе, независимо от того, делает ли пациент запись прямо сейчас или нет.

Когда пациент приходит записаться, администратор:
1. Берёт карточку пациента
2. Берёт расписание врача
3. Смотрит на свободные слоты
4. Создаёт запись

Сами объекты (карточка, расписание, запись) — стабильны и существуют независимо. Действие "записаться" — временное, оно использует эти объекты, но не определяет их форму.

Агрегат в DDD — это **объект** (или кластер объектов), а не действие.

---

## Часть 2. Что такое агрегат на самом деле

### Определение через три вопроса

Агрегат — это группа объектов, которая отвечает "да" на все три вопроса:

**Вопрос 1: Есть ли главный объект, без которого остальные теряют смысл?**

Слот в расписании (09:00–10:00 у доктора Иванова) — без расписания доктора Иванова этот слот бессмысленен. Значит расписание — главный объект (корень), а слот — подчинённый.

**Вопрос 2: Есть ли правила, которые применяются ко всей группе сразу?**

"В расписании не может быть двух слотов на одно время" — это правило применяется к расписанию как к целому. Нельзя проверить это правило, смотря только на один слот.

**Вопрос 3: Нужно ли сохранять всю группу вместе или можно по частям?**

Если добавляем слот в расписание — нужно проверить весь список слотов на конфликты и сохранить обновлённое расписание. Нельзя сохранить только новый слот, не зная об остальных.

Если все три ответа — "да", вы нашли агрегат.

### Корень агрегата (Aggregate Root)

Корень — это тот "главный объект" из вопроса 1. Его особенности:

- Имеет уникальный ID (по которому агрегат находят в хранилище)
- Является единственной точкой входа: чтобы что-то сделать с объектами внутри агрегата, нужно обращаться к корню
- Отвечает за соблюдение всех инвариантов агрегата

**Пример:**
```
Внешний мир хочет добавить слот → вызывает Schedule.AddSlot(time)
                                         ↑
                                    корень агрегата
                                    (не Slot.Create() напрямую)
```

Снаружи нельзя напрямую создать Slot — только через Schedule. Это защищает инварианты.

## Часть 3. Как найти корень агрегата — практический метод

### Шаг 1: Найдите все "вещи" в домене

Выпишите всё, о чём говорит бизнес. Не думайте пока о коде — просто перечислите понятия:
- Пациент, Специалист, Запись, Расписание, Слот, Клиника, Контакт, Специализация...

### Шаг 2: Проверьте каждый объект на идентичность

Задайте вопрос: **"Этот объект — тот же самый, если у него изменились все свойства?"**

- Пациент Иванов поменял телефон → он всё равно тот же пациент? **Да** → Entity
- Временной слот 09:00–10:00 изменился на 10:00–11:00 → это тот же слот? **Нет, это другой слот** → Value Object
- Запись №123 изменила статус с "создана" на "подтверждена" → это та же запись? **Да** → Entity

**Правило:** Если объект "тот же самый" даже при изменении свойств — это Entity. Если при изменении любого свойства это уже "другой объект" — это Value Object.

### Шаг 3: Сгруппируйте по инвариантам

Инвариант — правило, которое **всегда** должно быть истинным. Найдите инварианты в требованиях:

> "Два слота не могут пересекаться по времени"

Какие объекты нужно видеть одновременно, чтобы проверить это правило?
→ Нужен весь список слотов + новый слот
→ Значит, все слоты и их список — в одном агрегате

> "Пациент не может иметь две записи на одно время"

Какие объекты нужно видеть?
→ Все записи пациента + новая запись
→ Это много записей из разных мест → это **кросс-агрегатный** инвариант (см. Часть 5)

### Шаг 4: Проверьте "кто без кого не живёт"

Задайте вопрос: **"Если удалить корень — нужны ли ещё подчинённые объекты?"**

- Удалили расписание → нужны ли ещё слоты этого расписания? **Нет** → Слоты живут внутри Schedule
- Удалили специалиста → нужны ли ещё его записи? **Да, они нужны для истории** → Appointment живёт отдельно

### Шаг 5: Проверьте размер

**Агрегат должен быть маленьким.** Если при загрузке агрегата вы тянете десятки связанных объектов — граница проведена неверно.

Классический тест: *"Сколько строк из базы данных нужно загрузить, чтобы получить этот агрегат?"*

- Schedule с 20 слотами → 1 + 20 = 21 строка → нормально
- Patient с историей всех 500 записей → 1 + 500 = 501 строка → слишком много, это неправильная граница

## Часть 4. Связь агрегатов с use cases

### Use case НЕ определяет форму агрегата — он ИСПОЛЬЗУЕТ агрегаты

Use case (сценарий использования) — это последовательность шагов, которую выполняет application service. Application service:
1. Загружает нужные агрегаты из репозиториев
2. Вызывает методы на агрегатах
3. Сохраняет изменённые агрегаты
4. Публикует domain events

Агрегаты при этом не меняются — меняется только то, какие из них загружаются.

### Пример: Use case "Пациент записывается к специалисту"

```
Application Service: CreateAppointmentCommandHandler

1. Загрузить: Schedule (агрегат расписания специалиста)
2. Загрузить: Patient (агрегат пациента) — чтобы убедиться, что пациент существует
3. Проверить: есть ли у пациента уже запись в это время
   (через AppointmentRepository.FindByPatientAndTime)
4. Вызвать: schedule.BookSlot(slotId)  ← метод агрегата Schedule
5. Создать: новый Appointment(patientId, specialistId, timeSlot)  ← новый агрегат
6. Сохранить: Schedule (обновлённое)
7. Сохранить: Appointment (новое)
8. Опубликовать: AppointmentCreated event
```

Здесь Schedule и Appointment — отдельные агрегаты. Use case их объединяет на время выполнения, но они не "сливаются" в один агрегат.

### Пример: Use case "Пациент отменяет запись"

```
Application Service: CancelAppointmentCommandHandler

1. Загрузить: Appointment по appointmentId
2. Вызвать: appointment.Cancel(cancelledBy: patientId, cancelledAt: now)
   → внутри метода проверяется инвариант "не позже чем за 2 часа"
3. Загрузить: Schedule специалиста
4. Вызвать: schedule.ReleaseSlot(slotId)
5. Сохранить: Appointment (обновлённое)
6. Сохранить: Schedule (обновлённое)
7. Опубликовать: AppointmentCancelled event
```

Снова — два агрегата (Appointment и Schedule), объединённых только на время выполнения use case.

### Ключевое понимание

```
Use Case (Application Layer)          Domain Layer
───────────────────────────     ─────────────────────────────
                              
Команда: CancelAppointment ──→ Appointment.Cancel(by, at)
                                    ├── Проверяет инвариант
                                    ├── Меняет статус
                                    └── Публикует event
                          ──→ Schedule.ReleaseSlot(slotId)
                                    ├── Находит слот
                                    └── Меняет статус на Free
```

Use case — это дирижёр. Агрегаты — музыканты. Дирижёр не определяет, как устроена скрипка.

## Часть 5. Кросс-агрегатные инварианты — самое сложное

### Проблема

Инвариант "пациент не может иметь две записи на одно время" затрагивает несколько агрегатов Appointment. В DDD нельзя атомарно обновить два агрегата в одной транзакции (правило: один агрегат = одна транзакция).

### Решение 1: Проверка в Application Service (простое)

Перед созданием Appointment application service явно проверяет через репозиторий:

```
if (await appointmentRepo.HasConflict(patientId, timeSlot))
    throw new ConflictException("У пациента уже есть запись в это время");
```

Это работает, но не является "чистым" DDD — проверка не в домене, а в application слое. Приемлемо для большинства проектов.

### Решение 2: Eventual Consistency через Domain Events (правильное DDD)

```
1. Appointment создаётся (без немедленной проверки конфликта)
2. Публикуется event: AppointmentCreated(patientId, timeSlot)
3. EventHandler проверяет: есть ли конфликт?
4. Если есть → отменяет только что созданную запись автоматически
5. Пациент получает уведомление: "Запись отменена, выберите другое время"
```

Это "в конечном счёте согласованно" (eventual consistency) — на доли секунды конфликт существует, но сразу исправляется.

### Когда какое решение выбирать

| Ситуация | Решение |
|----------|---------|
| Конфликт недопустим даже на миллисекунду (банковские операции) | Pessimistic locking в Application Service |
| Небольшой период несогласованности допустим | Eventual Consistency через Events |
| Простой проект, нет распределённой системы | Проверка в Application Service |

## Часть 6. Как определить поля агрегата

### Алгоритм

**Шаг 1: Напишите все инварианты агрегата**

Для Appointment:
- Статус может переходить только по заданным путям (Created→Confirmed, Created→Cancelled, не обратно)
- Пациент может отменить только за 2+ часа до начала
- Отменённую запись нельзя подтвердить

**Шаг 2: Для каждого инварианта спросите: "Что нужно знать, чтобы проверить это правило?"**

| Инвариант | Что нужно знать |
|-----------|----------------|
| Переход статуса | Текущий статус → `AppointmentStatus` |
| 2 часа до начала | Время начала записи, текущее время → `TimeSlot`, `CancelledAt` |
| Нельзя подтвердить отменённую | Текущий статус → `AppointmentStatus` |

**Шаг 3: Всё что нужно для инвариантов → поля агрегата**

```
Appointment
  ├── AppointmentId       (для идентичности)
  ├── PatientId           (для знания "чья запись", нужен для отмены пациентом)
  ├── SpecialistId        (для контекста)
  ├── TimeSlot            (нужен для проверки "2 часа до начала")
  └── AppointmentStatus   (нужен для всех статусных инвариантов)
```

**Шаг 4: Всё остальное — НЕ поля агрегата**

- Имя пациента → не нужно для инвариантов Appointment → не в агрегате
- Рабочие часы специалиста → не нужны для инвариантов Appointment → не в агрегате
- Другие записи пациента → не нужны для инвариантов Appointment → не в агрегате

### Главный вопрос для каждого потенциального поля

> "Нужно ли это поле, чтобы проверить хотя бы один инвариант этого агрегата?"

Если **да** → включить в агрегат.
Если **нет** → не включать. Это либо принадлежит другому агрегату, либо нужно только для чтения (read model).

## Часть 7. Типичные ошибки и как их распознать

### Ошибка 1: "God Aggregate" — агрегат знает обо всём

**Признак:** При создании одного объекта нужно загрузить 5+ других агрегатов.

**Симптом в коде:**
```
// Это неправильно
var appointment = new Appointment(
    patient: fullPatientObject,           // зачем целый пациент?
    specialist: fullSpecialistObject,     // зачем целый специалист?
    allPatientAppointments: [...],        // зачем все записи?
    schedule: fullScheduleWithAllSlots,   // зачем всё расписание?
    ...
);
```

**Решение:** В агрегате хранить только ID (ссылки), а не полные объекты.

```
// Правильно
var appointment = new Appointment(
    patientId: patient.Id,           // только ID
    specialistId: specialist.Id,     // только ID
    timeSlot: selectedSlot           // только конкретный слот
);
```

---

### Ошибка 2: Агрегат = Use Case

**Признак:** Агрегаты называются как действия: "СозданиеЗаписи", "ОтменаЗаписи".

**Почему это ошибка:** Один и тот же объект (Appointment) участвует в разных use cases — создание, подтверждение, отмена, завершение. Если делать агрегат под каждый use case — будет дублирование и несогласованность.

**Правило:** Агрегат называется существительным (Appointment, Schedule). Use case — глаголом (CreateAppointment, CancelAppointment).

---

### Ошибка 3: Хранить вычисляемые данные как поля агрегата

**Пример:** Хранить `FreeSlots` в агрегате Schedule.

**Проблема:** При каждом бронировании нужно обновлять и список записей, и список свободных слотов. Два места хранения одной истины → риск рассинхронизации.

**Правило:** Если значение можно вычислить из других полей агрегата — это метод, а не поле.

```
// Неправильно:
schedule.FreeSlots  // хранимое поле

// Правильно:
schedule.GetFreeSlots()  // метод, вычисляет = AllSlots - BookedSlots
```

---

### Ошибка 4: Слишком маленькие агрегаты без инвариантов

**Пример:** Выделить каждый Slot как отдельный агрегат.

**Проблема:** Инвариант "нет пересекающихся слотов" теперь невозможно проверить — нужно видеть все слоты одновременно, а они в разных агрегатах.

**Правило:** Объекты, чьи инварианты зависят друг от друга, должны быть в одном агрегате.

---

### Ошибка 5: Коллекции сущностей как Value Objects

**Пример:** "Записи пациента" как Value Object.

**Проблема:** Value Object неизменяем, определяется своими значениями. Список записей — изменяемое состояние, каждая запись имеет свой ID. Это не VO.

**Правило:** Список сущностей — это коллекция внутри агрегата (если они принадлежат этому агрегату) или запрос через репозиторий (если это отдельные агрегаты).

## Часть 8. Быстрые эвристики (шпаргалка)

### Как определить тип объекта

```
Вопрос 1: "Этот объект имеет уникальную идентичность?"
  → Нет → Value Object
  → Да → Вопрос 2

Вопрос 2: "Этот объект может существовать без 'владельца'?"
  → Нет → Entity (внутри агрегата)
  → Да → Вопрос 3

Вопрос 3: "Другие агрегаты ссылаются на этот объект?"
  → Да → Aggregate Root
  → Нет → Entity (внутри агрегата)
```

### Как проверить границу агрегата

1. **Тест транзакции:** "Нужно ли сохранять это как одну атомарную операцию?" → Если да — в одном агрегате.

2. **Тест инварианта:** "Можно ли проверить все бизнес-правила, загрузив только этот агрегат?" → Если нет — граница неверна.

3. **Тест размера:** "Сколько записей из БД нужно загрузить?" → Если больше нескольких десятков — скорее всего агрегат слишком большой.

4. **Тест жизненного цикла:** "Все объекты внутри рождаются и умирают вместе?" → Если нет — возможно, это отдельный агрегат.

### Что хранить в агрегате, а что запрашивать

```
ВНУТРИ агрегата:
  ✓ Поля, нужные для проверки инвариантов
  ✓ Дочерние объекты без самостоятельного жизненного цикла
  ✓ Value Objects, описывающие состояние
  ✓ Типизированные ID (ссылки на другие агрегаты)

СНАРУЖИ (запрашивать через репозиторий):
  ✗ Полные объекты других агрегатов
  ✗ Коллекции, которые растут неограниченно
  ✗ Данные, нужные только для отображения (read model)
  ✗ Вычисляемые значения (сделать методом)
```

## Часть 9. Применение к нашей задаче

### Почему Schedule — Aggregate Root, а не просто Entity?

Проверим по признакам:
- Другие агрегаты ссылаются на него? → Appointment хранит SpecialistId, и через него находит Schedule → **да**
- Есть инварианты, затрагивающие несколько дочерних объектов? → "Слоты не пересекаются" затрагивает весь список слотов → **да**
- Слоты имеют смысл без расписания? → Слот "09:00-10:00" без расписания специалиста бессмысленен → **нет**

Вывод: Schedule = Aggregate Root, Slot = Entity внутри агрегата Schedule.

### Почему Patient — отдельный Aggregate Root, а не часть Appointment?

- Пациент существует независимо от записей? → **Да**, у него своя жизнь в системе
- Другие агрегаты ссылаются на пациента напрямую? → **Да**, несколько Appointment ссылаются на одного пациента
- Инварианты Appointment требуют знать полный объект Patient? → **Нет**, достаточно PatientId

Вывод: Patient = отдельный Aggregate Root, Appointment хранит только PatientId.

### Почему "Записи пациента" — не часть Patient?

Пациент может иметь 0 записей, 10, 100, 1000. Если хранить их внутри Patient:
- При загрузке пациента нужно загрузить все его 1000 записей
- При создании новой записи нужно загрузить всего пациента с 1000 записями, добавить одну запись, сохранить всё обратно
- Это нарушает принцип маленьких агрегатов

Правильно: Appointment — отдельный агрегат. Чтобы найти все записи пациента → `AppointmentRepository.FindByPatientId(patientId)`.

## Итоговая схема мышления при проектировании

```
1. Выпишите все понятия домена (существительные из требований)

2. Для каждого: Entity или Value Object?
   → Есть идентичность + жизненный цикл? Entity
   → Определяется только значениями? Value Object

3. Для каждой Entity: самостоятельный Aggregate Root или внутри другого?
   → На него ссылаются другие агрегаты? → Aggregate Root
   → Он не живёт без "владельца"? → Entity внутри агрегата-владельца

4. Для каждого Aggregate Root: какие поля?
   → Выпишите инварианты
   → Для каждого инварианта: что нужно знать для проверки?
   → Только это и идёт в поля

5. Найдите кросс-агрегатные инварианты (затрагивают несколько агрегатов)
   → Решите через Application Service (проверка перед операцией)
   → Или через Domain Events + Eventual Consistency

6. Назовите операции как методы агрегатов
   → Use cases = Application Service, вызывающий эти методы
   → Не путать use case с агрегатом
```
