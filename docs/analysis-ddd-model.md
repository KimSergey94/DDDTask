# Анализ доменной модели — Запись к специалисту

## Общая оценка

Работа показывает базовое понимание DDD: автор правильно использует термины, видит инварианты и старается обосновать решения. Однако есть несколько системных ошибок, которые важно исправить до написания кода — иначе архитектура будет неправильной с самого начала.

Главная проблема: **агрегаты спроектированы как операции (use cases), а не как кластеры объектов с общими инвариантами**. Это фундаментальное смешение понятий.

---

## Блок 1 — Entities (Сущности)

### Что верно

Пациент, Специалист — правильные сущности. У каждого есть уникальная идентичность, жизненный цикл и состояние, которое меняется независимо.

### Проблема 1: Расписание как сущность

**Расписание** — это не просто сущность, это **корень агрегата**. У расписания есть собственные инварианты (слоты не пересекаются, все слоты в рамках рабочих часов), есть набор дочерних объектов (сами слоты). Называть его просто "entity" недостаточно — надо явно указать, что это Aggregate Root.

### Проблема 2: Клиника — неопределённая роль

Клиника перечислена как сущность, но ни в одном агрегате не появляется. Непонятно:
- Что внутри клиники? Список специалистов? Расписания?
- Кто является корнем агрегата — клиника или специалист?

Если клиника — это просто способ сгруппировать специалистов, возможно, это вообще не нужная в этом домене абстракция. Либо это отдельный bounded context ("Управление персоналом"), который пока не входит в задачу.

### Проблема 3: Запись на приём — это сущность, а не только "слоты"

В секции entities написано "Запись на прием (Слоты)" — это смешение двух понятий:
- **Слот** — временной интервал в расписании специалиста (value object или дочерняя сущность внутри Schedule)
- **Запись на прием (Appointment)** — факт того, что конкретный пациент забронировал конкретный слот (это отдельная сущность со своим жизненным циклом: создана → подтверждена → отменена → завершена)

### Проблема 4: Администратор не указан как entity

Администратор появляется в агрегате 3, но отсутствует в списке сущностей. Если администратор участвует в доменных операциях (отменяет записи), он должен быть либо отдельной сущностью, либо ролью пользователя. Это надо явно отразить.

## Блок 2 — Value Objects

### Что верно

- **Контактные данные пациента** (телефон/email) — классический value object. Нет идентичности, определяется значением, неизменяем. Верно.
- **id Специалиста, id Пациента** — типизированные идентификаторы как value objects — это правильный подход (не `Guid`, а `SpecialistId`, `PatientId`). Верно.
- **Рабочие дни и рабочие часы** — верно. Это описание "как устроено время работы специалиста", не имеет собственной идентичности, определяется набором значений.

### Проблема 5: "Свободные слоты" — это не value object, это вычисляемое состояние

**Свободные слоты для записи** — это результат запроса: `все слоты расписания минус уже занятые`. Это не самостоятельный объект домена, который хранится и имеет смысл сам по себе. Это read-model или проекция, которую возвращает метод расписания.

Если хранить "свободные слоты" как отдельный VO, при каждой записи придётся синхронизировать два места — источник истины (список записей) и этот VO. Это нарушение Single Source of Truth.

### Проблема 6: "Записи пациента" — это не value object

**Записи пациента** — это коллекция сущностей (Appointment), связанных с пациентом. Это либо:
- Часть агрегата Patient (список `appointmentIds` — ссылок, не полных объектов)
- Либо запрашивается через репозиторий при необходимости

Value object — это объект без идентичности, неизменяемый, определяемый только своими свойствами. Коллекция записей — изменяемое состояние с историей, у каждой записи есть свой ID. Это не VO.

## Блок 3 — Агрегаты (главные ошибки)

### Критическая ошибка: Агрегаты 2–5 — это не агрегаты

> "Пациент отменяет запись", "Специалист отменяет запись", "Специалист блокирует слот", "Подтвердить запись"

Это **use cases / команды / операции**, а не агрегаты. Агрегат — это **кластер объектов с общими инвариантами и единой точкой изменения**. Отмена записи — это операция, которая выполняется **над** агрегатом Appointment.

**Как должно быть:**
```
Агрегат: Appointment (Запись на приём)
Операции над ним:
  - Cancel(by: Patient)      → проверяет инвариант "не позже чем за 2 часа"
  - Cancel(by: Specialist)   → другие правила
  - Confirm()                → проверяет, что статус позволяет подтверждение
```

Операции — это методы агрегата или команды в application service, но не отдельные агрегаты.

### Критическая ошибка: Агрегат "Запись на прием" слишком большой и нарушает границы

В первом агрегате перечислено:
- Пациент (сущность)
- Рабочие часы специалиста
- id Специалиста
- Свободные слоты специалиста
- Записи пациента
- Контактные данные пациента

Это означает, что при создании одной записи нужно загрузить:
1. Полный объект Пациента
2. Всё расписание специалиста с рабочими часами
3. Все существующие записи специалиста (чтобы найти свободные слоты)
4. Все существующие записи пациента (чтобы проверить конфликт)

Это нарушает **главный принцип агрегатов в DDD: загружать и сохранять один агрегат за транзакцию**. Такой "агрегат" превращается в транзакцию по всей базе данных.

**Правило Вернона (из "Implementing DDD"):** агрегаты должны быть маленькими. Если вы включаете в агрегат чужие объекты — это сигнал, что граница проведена неверно.

### Проблема: Кросс-агрегатный инвариант

Инвариант "пациент не может записаться на одно и то же время к разным специалистам" — это **кросс-агрегатный инвариант**. Он затрагивает несколько агрегатов (несколько Appointment объектов разных специалистов).

В DDD это решается одним из двух способов:
1. **Eventual consistency** — создаём запись, публикуем domain event `AppointmentCreated`, обработчик проверяет конфликты и при необходимости отменяет запись с причиной
2. **Pessimistic locking на уровне application service** — перед созданием записи явно проверяем через репозиторий, есть ли у пациента запись в это время

Нельзя включить "все записи пациента" внутрь агрегата создания записи — это нарушение границ.

## Блок 4 — Инварианты

### Что верно

- Слоты не пересекаются — правильный инвариант расписания
- Отменённую запись нельзя повторно подтвердить — правильный инвариант статусной машины Appointment
- Пациент может отменить только за 2+ часа — правильный бизнес-инвариант

### Проблема: Инвариант "блокировка слота отменяет все записи" реализован верно концептуально, но объяснение размыто

Написано: "все записи на этот период считаются отменёнными автоматически (порождается domain event)". Это правильный подход — через domain event. Но нужно уточнить:
- Кто публикует event? Агрегат Schedule (или Specialist) при блокировке слота
- Кто обрабатывает event? Application service или domain event handler, который находит все Appointment в этом периоде и отменяет их
- Это не "автоматически внутри агрегата" — это через event-driven взаимодействие между агрегатами

## Итог: Оценка по пунктам

| Аспект | Оценка | Комментарий |
|--------|--------|-------------|
| Выделение entities | Частично верно | Расписание недооценено, Администратор пропущен |
| Value objects | Частично верно | IDs, ContactInfo, WorkingHours — верно; "Свободные слоты" и "Записи пациента" — ошибка |
| Агрегаты | Неверно | Агрегаты 2–5 — это use cases, не агрегаты |
| Границы агрегатов | Неверно | Агрегат 1 слишком большой, нарушает принцип "один агрегат за транзакцию" |
| Инварианты | В основном верно | Правильные правила, неверная привязка некоторых из них |
| Domain Events | Упомянуты верно | Концепция схвачена, детали требуют доработки |

## Что изучить для исправления

1. **Aggregate Design** — Vernon "Implementing Domain-Driven Design", глава про агрегаты: размер агрегата, принцип "одна транзакция — один агрегат"
2. **Aggregate vs Use Case** — агрегат отвечает на вопрос "что это?", use case — "что с этим делают?"
3. **Кросс-агрегатная согласованность** — eventual consistency через domain events
4. **Value Object vs Entity** — ключевое отличие: наличие идентичности и изменяемости
