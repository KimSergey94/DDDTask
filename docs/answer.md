1.1) 
entity:
1) Клиника
2) Пациент
3) Специалист
4) Расписание
5) Запись на прием (Слоты)

value objects:
1) рабочие дни и рабочие часы Специалиста
2) Свободные слоты для записи
3) id Специалиста - скорее всего нужно для записи на прием, чтобы указать Специалиста
4) Записи пациента
5) Контактные данные пациента (телефон или email)
6) id Пациента

агрегаты:
1) Запись на прием:
1.1) Пациент сущность
1.1) рабочие дни и рабочие часы Специалиста - чтобы можно было проверить что не запись не выходит за рамки рабочего времени
1.2) id Специалиста
1.3) Свободные слоты Специалиста для записи - возможно будет нужно для выбора слота пациенту? но тут как и с рабочими часами чтобы запись была сделана в свободный слот и не выходила за рамки рабочего времени
1.4) Записи пациента (занятые слоты) - чтобы не записали на одно и то же время (даже к разным специалистам)
1.5) Контактные данные пациента - если не хранятся внутри сущности
1.6) Инвариант: Два слота не могут пересекаться
1.7) Инвариант: пациент не может иметь две записи на одно и то же время (даже к разным специалистам).

2) Пациент отменяет запись
2.1) Пациент сущность
2.2) Id записи, слот (value object)
2.3) Инвариант: не позже чем за 2 часа до приёма (После - только специалист или администратор)

3) Специалист/Администратор отменяет запись
3.1) Специалист/Администратор сущность
3.2) Id записи, слот (value object)

4) Специалист может заблокировать слот
4.1) Специалист сущность
4.2) Инвариант: все записи на этот период считаются отменёнными автоматически (порождается domain event)

5) Подтвердить запись (Пациент?)
5.1) Сущность пациента
3.2) Id записи, слот (value object)
5.2) Инвариант: Отменённую запись нельзя повторно подтвердить - только создать новую
